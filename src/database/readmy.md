# Структура бази даних та ключові характеристики системи

## 1. Таблиця users — користувачі системи

Зберігає інформацію про всіх людей, які мають доступ до системи.

### Поля

id — первинний ключ

first_name, last_name

phone

email (унікальний)

password (хешований)

role — driver, manager, admin

status (опціонально) — active, pending, blocked

created_at, updated_at

Призначення
визначає рівень доступу

використовується для авторизації

дозволяє масштабувати систему (самореєстрація, підтвердження адміном)

## 2. Таблиця cars — транспортні засоби

Використовується для прив’язки маршруту до конкретного автомобіля.

Поля
id

brand, model

plate_number

status — active, maintenance, inactive

timestamps

Призначення
контроль доступних авто

можливість розширення (ремонти, техогляди, пробіг)

## 3. Таблиця routes — маршрути

Центральна таблиця логістики, що описує роботу водіїв.

Поля
id

date

car_id (FK → cars)

driver_id (FK → users)

start_location, end_location

route_number

planned_km, planned_hours

actual_km, actual_hours

helper_name

notes

status — життєвий цикл маршруту:

planned

in_progress

completed

pending_approval

approved

cancelled

deleted

timestamps

Призначення
зберігає повний цикл маршруту

розділяє планові та фактичні дані

забезпечує контроль менеджера над водієм

## 4. Таблиця route_logs — історія змін

Фіксує всі дії з маршрутом для прозорості та аудиту.

Поля
id

route_id

user_id

action — create, update, status_change, delete

old_value, new_value

timestamps

Призначення
аудит

відстеження помилок

прозорість для менеджера та адміна

історія змін маршруту

Основні характеристики роботи з базою

### 1. Чітка рольова модель

Ролі визначають доступ:

Driver — оновлює фактичні дані, змінює статуси in_progress / completed.

Manager — створює маршрути, оновлює планові дані, підтверджує або відхиляє.

Admin — повний контроль: користувачі, авто, видалення маршрутів.

Це забезпечує безпеку та передбачуваність.

### 2. Статусна логіка маршрутів

Жорстко контрольована схема переходів:

Код
planned → in_progress → completed → pending_approval → approved
Додаткові стани:

cancelled — менеджер відхиляє

deleted — адмін видаляє

Це запобігає хаосу та неправильним змінам.

### 3. Розділення планових і фактичних даних

Планові дані → менеджер

Фактичні дані → водій

Підтвердження → менеджер

Це дозволяє контролювати якість даних і уникати маніпуляцій.

### 4. Логи як основа прозорості

Кожна зміна фіксується:

хто

коли

що змінив

старе значення

нове значення

Це дає:

аудит

історію

відповідальність

можливість розбору проблем

Важливі моменти, які потрібно зафіксувати
Email унікальний — основа авторизації.

Пароль завжди хешується — безпека.

Статуси маршруту — строго контрольовані — не можна перескочити етап.

Водій не може змінювати маршрут після approved — фінальний захист.

Менеджер не може видаляти маршрути — тільки адмін.

Логи — обов’язкові — для прозорості та аудиту.

Можливість додати самореєстрацію — без порушення логіки.

Можливість додати статус користувача pending — для майбутнього масштабування.
